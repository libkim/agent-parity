이 메모리는 사용자가 명시적으로 확정한 agent-parity 설계 불변 조건과, 이 작업에서 반복된 잘못의 합집합이다. 이후 작업에서는 기억으로 재구성하지 말고 이 기준과 사용자 원문, 실제 릴리스 이력, 현재 코드를 대조한 뒤 판단한다.

## 정확한 핵심 문제 번호

1. Claude SessionStart 훅의 bash 하드코딩.
2. Unix 관리 래퍼의 임시파일 cleanup 문제.
3. 설치된 버전이 아니라 main 설치 스크립트를 실행하는 문제.
4. 모든 플랫폼 MCP 바이너리를 사용자 프로젝트마다 vendoring하는 문제.
5. Windows CRLF 메모리 파일의 frontmatter 파싱 실패 문제.
6. cross-OS MCP 설정 불일치와 self-heal 문제. 설정이 실제 변경됐을 때 세션 재시작을 안내하는 정책도 문제 6에 포함된다.
7. 여러 출시 버전을 건너뛴 업데이트에서 훅 문자열 기반 마이그레이션이 과거 형식을 놓칠 수 있는 문제.

번호가 붙은 문제 목록을 말하거나 기록할 때는 위 목록을 임의로 재배열·분할·대체하지 않는다. self-heal은 문제 5가 아니라 문제 6이다. `.claude/settings.json` 부트스트랩은 이 번호 목록의 문제 6이 아니다.

## 설계와 책임 경계

- 명시적으로 확정된 설계를 불변 조건으로 취급한다. install과 update를 다시 결합하지 않는다. sync-claude와 self-heal을 다시 결합하지 않는다. 이미 정한 명령 이름과 책임 경계를 임의로 바꾸지 않는다.
- 기능 범위와 명령 이름을 일치시킨다. Claude 전용 동기화만 `sync-claude`이고, 공통 기능은 인자 없는 `self-heal`이다. 공통 self-heal에 claude 같은 에이전트 이름을 붙이지 않는다.
- 원격 `install`과 원격 `update`는 별도 진입점이다. 설치 전에는 프로젝트 로컬 공통 코드가 없으므로 부트스트랩 경계의 일부 중복은 의도적으로 허용한다. 잘못된 DRY로 합치지 않는다.
- 로컬 관리 명령 `status`, `version`, `uninstall`, `sync-claude`, `self-heal`은 서로 분리된 로컬 스크립트다. `update`만 최신 릴리스의 원격 updater를 선택한다.
- 핵심 설정 처리와 에이전트별 훅 표현 계층을 분리한다. 설정 수정 로직에 불필요한 에이전트 인자와 출력 어댑터를 넣지 않는다.
- 소스 템플릿과 이미 설치된 프로젝트 산출물의 소유권을 구분한다. 템플릿 변경을 이유로 현재 프로젝트의 설치 산출물을 직접 바꾸지 않고 정식 update 경로로만 교체한다.
- 원격 소스, GitHub Release Asset, 사용자 공용 캐시, 프로젝트 로컬 파일을 설명과 구현에서 항상 구분한다.

## 소유권과 사용자 파일 보존

- 부분 문자열은 소유권 증명이 아니다. `sync-claude`나 `self-heal`이 들어 있다는 이유만으로 훅을 소유했다고 판단하지 않는다. Cursor CLI에 특정 문자열이 있다는 이유로 파일 전체를 소유했다고 판단하지 않는다.
- 소유권은 정확한 명령의 완전 일치, 명시적 마커, 실제 배포 파일의 완전 일치나 해시처럼 검증 가능한 근거로만 판단한다.
- 사용자 파일 보존은 일부 기능이 아니라 모든 수정·삭제 경로의 시스템 불변 조건이다. MCP 설정, Claude 설정, hooks, Cursor CLI, AGENTS.md, gitignore 모두 같은 원칙을 적용한다.
- 소유권을 증명할 수 없으면 덮어쓰기·삭제하지 않고 변경하지 않거나 수동 조치를 안내한다.

## 실제 이력과 마이그레이션

- 실제 릴리스 이력이 아닌 상상한 과거 상태를 호환 대상으로 만들지 않는다. 잠깐 로컬에 존재했지만 출시되지 않은 `self-heal claude`, 가짜 `/old/sync-claude.sh`, 출시되지 않은 파일명을 마이그레이션·테스트·제거 대상으로 등록하지 않는다.
- 마이그레이션을 정상 실행 경로의 범용 문자열 추측으로 누적하지 않는다. 실제 출시 버전 사이의 명시적이고 순차적이며 멱등적인 변환으로 별도 관리한다.
- 테스트 입력은 실제 릴리스, 현재 명세, 사용자가 명시적으로 지원하기로 한 상태에서만 가져온다. 구현을 정당화하기 위한 가상 과거 상태를 테스트에 넣지 않는다.

## 런타임과 네트워크 경계

- 로컬 설정 수정과 로컬 관리 기능을 MCP 서버 바이너리에 결합하지 않는다. `status`, `uninstall`, `sync-claude`, `self-heal`은 MCP 런처나 바이너리를 실행하거나 캐시 다운로드를 유발하지 않는다.
- `status`의 네트워크 접근은 최신 버전 확인 필드에만 허용하며, 실패하면 그 필드만 `unknown (network unavailable)`로 표시한다.
- `uninstall`은 완전히 오프라인이어야 한다.
- 프로젝트의 무의존성 원칙과 충돌하는 Python, Node, Go 같은 새 런타임 의존성을 사용자 승인 없이 추가하지 않는다. Unix uninstall에 Python 3을 요구하도록 만든 현재 변경은 재검토가 필요한 잘못이다.

## 구현·검증 절차

- 개념, 설정 파일의 소유 주체, 실제 파일 실행 흐름을 먼저 확정하고 구현한다. 파일 흐름을 설명할 때 추상 개념을 파일 경로 사이에 섞지 않는다.
- Git Bash, native CMD, PowerShell의 실행 흐름을 추측하지 말고 실제 파일과 셸 규칙으로 검증한다.
- 완료라고 말하기 전에 해당 문제의 전체 호출 경로를 확인한다. 한 함수나 부분 테스트 통과를 문제 전체 완료로 표현하지 않는다.
- 문제별 체크포인트를 유지해 어떤 코드가 어느 문제 때문에 추가됐는지 추적한다. 여러 문제를 근거 없이 하나의 누적 변경으로 섞지 않는다.
- 설명할 때 실행 위치와 파일 위치를 생략하지 않는다. 공유 함수가 원격 스크립트인지, 프로젝트 로컬 스크립트인지, MCP 바이너리인지 명시한다.

## 이 작업에서 반복된 판단 실패

- 사용자 원문과 코드를 확인하지 않고 빈 부분을 추측으로 채운 뒤 사실처럼 단정했다.
- install/update, sync-claude/self-heal, 런처/스크립트의 이미 확정된 경계를 반복해서 다시 결합했다.
- 존재하지 않은 과거 명령과 파일을 만들고 테스트로 정당화했다.
- 부분 문자열을 소유권으로 오인해 사용자 파일 전체 삭제 위험을 만들었다.
- self-heal만 로컬로 바꾼 뒤 status와 uninstall의 MCP 바이너리 의존성을 확인하지 않고 완료라고 했다.
- Cursor CLI 제거에 다른 설정과 같은 소유권 보호 규칙을 적용하지 않았다.
- 번호 목록을 검증하지 않고 문제 5를 self-heal로 바꿔 말했으며, 잘못된 `issue-5` 메모리까지 생성했다. 지적받은 뒤에도 `.claude` 부트스트랩을 문제 6이라고 다시 추측했다.
- CRLF 파서 결함 때문에 손상된 메모리를 먼저 검증하지 않고 정상 근거로 사용했다. Windows CRLF frontmatter 문제는 핵심 문제 5다. `store.go`는 CRLF를 LF로 정규화해 정상 입력을 파싱하도록 수정했다. 작업 중 생긴 중첩 손상 형태를 제품의 자동 복구 대상으로 추가한 것은 가상 과거 상태를 영구 호환 코드로 만든 동일한 잘못이어서 제거했다. 손상된 기존 메모리 6개만 원래 created·tags와 누적 strength·lastAccessed를 합쳐 물리 복구했다.
- 사용자가 이미 제공한 잘못 목록을 보존하지 않고 더 불완전한 새 목록으로 대체했다.

앞으로 불확실한 내용은 확정적으로 말하거나 메모리에 사실로 기록하지 않는다. 먼저 사용자 원문, 이 기준 메모리, 실제 릴리스 태그·커밋, 현재 호출 경로를 대조한다. 충돌하면 사용자에게 충돌을 명시하고 임의로 번호나 설계를 만들어내지 않는다.
## 설명 기준 고정 규칙 (2026-07-19)

- 사용자가 실행 흐름 비교를 요구하면 먼저 비교 기준을 하나로 고정하고 끝까지 바꾸지 않는다. 파일 흐름을 묻는 경우 실제 파일만 화살표에 쓰며, 추상 개념이나 다른 분류 기준을 중간에 끼우지 않는다.
- 여러 명령을 묶거나 분리했다면 그 기준을 정확히 말한다. `status/version/uninstall`을 함께 쓴 이유는 동일 실행 파일이어서가 아니라 `project-agent-parity.sh`의 동일한 `case` arm(`uninstall | status | version`)을 공유하기 때문이다. `update`는 별도 `case` arm이지만, Git Bash에서 Windows 분기가 빠졌다는 결함을 설명할 때는 네 명령 모두 같은 원인과 결과이므로 불필요하게 분리하지 않는다.
- 근거를 확인하지 않은 채 사후 설명을 만들지 않는다. 이번 대화에서 “실행 파일이 달라서 분리했다”라고 말한 것은 사실과 모순되는 허위 설명이었다.
- 약속이나 재발 방지 규칙을 말하기 전에 실제로 영구 메모리에 기록했는지 확인한다. 기록하지 않았다면 기록했다고 암시하지 말고, 먼저 저장한 뒤 저장 위치를 명시한다.

## 릴리스 부트스트랩과 전역 호출 경로 감사에서 확인한 재발 (2026-07-19)

- 새 릴리스 asset 설치 경로를 추가하면서 기존 `install`의 latest 조회 fallback을 제거하지 않았다. 새 책임 경계를 만들었으면 대체된 정상 실행 경로와 fallback을 전역 검색해 제거해야 하며, 지원·테스트하기로 명시하지 않은 직접 실행 경로를 편의상 남기지 않는다.
- `agent-parity-config`로 로컬 설정 처리를 분리했다고 말했지만 전체 호출 경로를 확인하지 않았다. 실제 `install.sh`, `update.sh`, `install.ps1`, `update.ps1`은 훅과 Claude 설정을 합칠 때 여전히 `run.sh`/`run.cmd`를 통해 `memory-mcp`를 실행한다.
- 그 결과 MCP 서버 본체 `main.go`에 설정 편집 플래그가 남고 `config.go`가 MCP 바이너리에 함께 빌드된다. 로컬 설정 편집은 config editor만 담당하고 memory MCP는 네 메모리 도구와 버전 출력만 담당해야 한다.
- Windows에서도 config editor로 설정 파싱을 통일한다고 설명했지만 `templates/common.ps1`에 JSON 훅·Claude 설정 제거 로직이 별도로 중복돼 있었다. 구현을 확인하지 않은 채 의도한 설계를 현재 사실처럼 설명하지 않는다.
- 설치·업데이트가 `run.* -version`을 실행해 MCP 바이너리를 미리 다운로드하면서 README에는 MCP 첫 사용 시 다운로드한다고 적혀 있다. 구현, 테스트, 문서의 생명주기 설명을 반드시 같은 호출 경로로 대조한다.
- 릴리스 CI가 Go 테스트만 실행하고 저장소의 shell uninstall 테스트, install/update 통합 테스트, PowerShell 구문·동작, Git Bash 분기를 검증하지 않는다. 수동 테스트 한 번을 릴리스 회귀 방지로 취급하지 않는다.
- `CLAUDE.md` 정확 일치 래퍼를 MCP 설정 순회 목록에 섞어 status에서도 MCP 등록처럼 출력한다. 파일의 책임과 출력 분류를 일치시키고, 단지 반복을 줄이기 위해 다른 개념을 같은 목록에 넣지 않는다.
- 명시적 begin/end 마커는 둘 다 있고 순서와 범위가 유효할 때만 소유 블록 증거다. 한쪽만 있는 상태에서 새 블록을 덧붙이거나 제거하지 말고 손상 상태를 보고해야 한다.
- Unix/Go 경로에 원자적 임시파일 교체를 적용했으면 같은 사용자 파일을 쓰는 PowerShell 경로에도 동일한 보존 규칙을 적용한다. 플랫폼 하나만 고친 뒤 전역 완료라고 말하지 않는다.

## 앞으로의 핵심 개발 방향 (2026-07-19)

agent-parity의 다음 개발은 아래 두 축을 같은 우선순위로 추진한다.

1. **발견된 불필요·중복·추측성 코드의 제거와 최적화**
   - 실제 출시 이력이나 현재 명세에 없는 호환 코드, fallback, 별칭, 마이그레이션을 만들지 않는다.
   - 전체 호출 경로를 확인하여 중복 책임, 잘못된 결합, 불필요한 런타임·네트워크·MCP 의존성을 제거한다.
   - 코드를 추가하는 국소 봉합보다 원인이 된 책임 경계와 실행 경로를 단순화한다.

2. **규칙을 실제로 강제하는 하네스의 고도화**
   - 필수 설계 규칙을 memory에만 의존하지 않고 저장소의 추적 문서와 AGENTS.md 진입점에서 항상 발견할 수 있게 한다.
   - 세션에 memory 도구가 노출되지 않는 경우와 MCP 서버 자체가 미등록·고장인 경우를 구분해 진단한다.
   - 수정 전 불변 조건 확인, 전체 호출 경로 감사, 플랫폼별 테스트, 릴리스 차단 검사를 자동화한다.
   - 규칙 준수를 모델의 회상이나 약속에만 맡기지 않고 hooks·구조 테스트·CI로 검증한다.

`memory_recent`, `memory_add`, `memory_search`, `memory_get`이 현재 호출 가능한 도구 표면에 보이지 않는다는 사실만으로 memory MCP가 등록되지 않았거나 기능 자체가 없다고 단정하지 않는다. 도구 노출 상태, MCP 등록 상태, 서버 시작 상태, 개별 도구 호출 가능성을 각각 분리해 확인한다.
