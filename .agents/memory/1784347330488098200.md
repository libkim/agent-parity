agent-parity 문제 2 재검토 결과: 원래 결함인 Unix 프로젝트 런처의 임시 update.sh 실행 시 `exec` 때문에 EXIT trap이 실행되지 않던 문제는 해결됐다. 현재 templates/project-agent-parity.sh는 `sh "$tmp"`를 자식 프로세스로 실행하고 성공·실패 후 명시 삭제하며 EXIT/signal trap도 둔다. 그러나 문제 2를 임시·스테이징 파일 정리 전체로 보면 아직 부분 해결이다. install.sh/update.sh의 `*.agent-parity.tmp`, update.sh의 `agent-parity.new`, Windows의 `agent-parity.cmd.new`는 중단·실패 시 잔존 가능하며, Unix 런처의 `${TMPDIR:-/tmp}/agent-parity-update.$$`도 mktemp가 아닌 예측 가능한 이름이다. install/update의 바이너리 직접 최종 경로 다운로드도 실패 시 부분 파일 위험이 있어 원자적 temp→검증→rename이 필요하다. 따라서 issue 2는 2a(exec/trap)는 완료, 2b(전역 temp/staging cleanup·atomicity)는 미완료로 관리한다.
