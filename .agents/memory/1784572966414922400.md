사용자 제안 설계 후보(2026-07-21): 클라우드 동기화 충돌을 append-only 이벤트로 봉합하기 전에, `strength`와 `lastAccessed`를 메모리 Markdown의 공유 가변 상태로 유지할 필요 자체를 재검토한다. 목적은 자주 쓰는 기억을 검색에서 더 잘 회상하는 것이므로, 본문 파일은 읽기 시 수정하지 않고 검색/필터링 시 현재 시각과 정적·로컬 신호를 조합하는 근본 변경을 고려한다. 검토 시 구분할 것: 현재 시각은 오래됨(decay)만 계산할 뿐 사용 빈도를 알려주지 않으며, 파일 mtime은 복사·동기화 서비스에 따라 바뀌므로 정본으로 부적합하고 인터넷 시간은 불필요한 네트워크 의존성을 만든다. 후보 구조는 immutable created timestamp + 명시적 pin/priority는 공유 Markdown, recall frequency와 lastAccessed는 장치별 로컬 SQLite/cache, semantic relevance와 age decay는 검색 시 계산하는 방식이다. 이 경우 장치 간 사용 이력을 공유하지 않는다는 제품 선택이 필요하다. 장치 간 학습 공유가 필수일 때만 append-only recall event를 검토한다.
