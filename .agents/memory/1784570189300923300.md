클라우드 동기화 전송 조사 결론(2026-07-21): 파일 탐색기 overlay merge는 지원 전제에서 제외하고, bidirectional sync를 별도 모드로 다룬다. 공식 Obsidian Sync는 `.obsidian`을 제외한 dot-prefixed 파일/폴더를 항상 제외하므로 현재 `.agents`, `.codex`, `.cursor`, `.claude`, `.mcp.json` 및 `.agents/memory` 중심 구조를 완전 동기화할 수 없다. Google Drive류 일반 파일 동기화는 파일을 운반할 수 있지만 의미적 JSON/TOML/YAML 병합이나 프로젝트 단위 트랜잭션을 제공하지 않는다. 현재 cross-OS 설계에서는 Windows self-heal이 run.cmd를 기록하고 Unix self-heal이 run.sh를 기록하므로 양쪽 장치가 동시에 활성화되면 네 MCP 설정이 왕복(ping-pong)하고 세션 시작 시 반대 OS 값이 보이면 MCP 로드 실패와 재시작 안내가 반복될 수 있다. 한 장치씩 사용하고 동기화 완료 후 전환해도 OS 전환 첫 세션의 교정·재시작은 남는다. 메모리는 새 ID 추가는 대체로 병행 가능하지만 동일 메모리의 strength/lastAccessed/tags/body 동시 변경은 Git merge driver가 없는 동기화 서비스에서 last-writer-wins, 충돌 복사본, 또는 손상된 frontmatter가 될 수 있다. 현재 메모리 write가 직접 os.WriteFile이라 sync client가 부분 기록을 관측할 위험도 있다. 기본 계층 후보는 (1) OS별 command를 동기화 파일 밖의 machine-local derived state로 분리하거나 OS 중립 명령을 마련해 ping-pong 제거, (2) 메모리 원자적 temp+rename 쓰기, (3) 충돌 복사본 탐지와 애플리케이션 수준 결정적 병합, (4) 여러 배포 파일의 혼합 세대를 감지하는 manifest/generation 검증이다. Unix 실행권한 보존 여부는 서비스/클라이언트별 실측이 필요하며 확정 사실로 취급하지 않는다.
