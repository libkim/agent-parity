폴더 전송 방식 조사(2026-07-21): `새/빈 대상에 완전한 프로젝트 폴더 복사`와 `기존 프로젝트 폴더 위에 파일 탐색기 덮어쓰기`를 같은 동작으로 취급하면 안 된다. 현재 zero-install cross-OS 테스트는 전자에 가까우며, 반대 OS의 run.sh/run.cmd 설정과 빈 캐시에서 self-heal이 네 MCP 설정을 현재 OS로 교정하고 바이너리를 준비한 뒤 재시작을 안내하는 경로를 검증한다. 후자의 Explorer overlay는 의미적 merge나 트랜잭션이 아니라 source에 있는 파일별 교체이며 destination-only 파일은 남을 수 있다. 따라서 (1) 동일 경로의 사용자 JSON/TOML/AGENTS 설정은 source가 통째로 덮어써 destination 변경이 self-heal 전에 이미 손실될 수 있고, (2) retired destination-only 산출물은 남는데 tombstone 정리는 install/update에만 있어 self-heal은 제거하지 않으며, (3) 중단/잠금/건너뛰기로 VERSION·RELEASE·scripts/hooks가 서로 다른 릴리스인 혼합 상태가 가능하고, (4) 같은 ID의 memory 파일은 Git merge driver 없이 last-writer-wins로 한쪽 strength/tags/body가 손실될 수 있으며, (5) .git 디렉터리까지 기존 repo 위에 merge-copy하면 Git metadata 혼합 위험이 있다. 제품 거버넌스/테스트는 transport mode를 fresh copy, overlay replacement, bidirectional sync로 분리하고 지원 보장과 충돌 규칙을 각각 정의해야 한다. self-heal은 완전 복사 후 OS 명령 교정에는 유효하지만 외부 copier가 만든 데이터 손실·stale files·mixed release를 일반 복구하지 않는다.
